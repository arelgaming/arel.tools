What Is Arel

Arel is a World of Warcraft raid management platform. It plugs directly into Discord, putting scheduling, roster, strats, and attendance in one place — so your team spends less time on admin and more time pulling.
The four core product areas:
Scheduling & SignupPost raid nights, set lockout windows, collect signups via web or Discord. Recurring schedules and automatic reminders available on Supporter plan.
RosterManage specs, roles, and availability. Build your roster from signups. Bench and standby management built in. Automated roster suggestions on Supporter plan.
Encounter PlanningShare strats and assign roles per encounter. Accessible on web and in-game via addon before the pull.
Attendance &
Bench Tracking		Attendance pulled automatically from Warcraft Logs — no manual marking after raids. Per-raider history persists across weeks. Bench and standby appearances tracked alongside raid appearances for roster decisions and trial raider evaluation.
Each guild operates in its own fully isolated space. The Discord guild is the tenant unit — membership, roles, and permissions flow from Discord into Arel.

Deployment Phases

Two deployment topologies — launch configuration and production split. The architecture is identical in both; only the process boundary changes.
The rest of this document describes the Phase 2 (Split) topology — three separate containers with private networking. Phase 1 collapses those containers into one process to minimise cost and operational overhead at launch. The code architecture, schema, procedure hierarchy, and all other decisions are identical in both phases. Splitting is a deployment config change plus one code change — not a migration.
Phase 1 — Launch (Single Process)
apps/api, apps/bot, and the Inngest worker handler run as a single Node.js process in one container. The bot calls tRPC procedures in-process (direct function call — no HTTP hop). All import boundaries and business logic rules still apply — the monorepo package structure enforces them at the code level regardless of process topology.
Container hostRailway or Fly.io — one container, one deploy.
VercelHobby tier — free. Custom domains now supported.
NeonFree tier — autosuspend after 5 min inactivity. Cold start ~500ms–1s. Acceptable at low guild count; upgrade to Launch ($19/mo) when cold starts become a user complaint.
FlagsmithCloud free tier (up to 1000 flags). Eliminates a container and a DB. Swap to self-hosted in Phase 2 with a one-line config change in packages/config.
Upstash RedisFree tier (10k requests/day). Sufficient for low guild count.
Service tokensSkip inter-service token verification in Phase 1 — everything is in-process. Wire it in during Phase 2 when services become separate HTTP callers.
Estimated cost~$5–15/mo (Railway/Fly usage + optional Neon upgrade).
Phase 2 — Split (Production Topology)
Three separate containers on Northflank with private networking. Bot and worker call tRPC over HTTP via TRPC_API_URL env var. Flagsmith self-hosted. This is the topology described in the rest of the document.
Container hostNorthflank — private network, per-service scaling, preview container spin-up via GitHub Actions.
VercelPro tier ($20/mo) — team seats, higher bandwidth limits.
NeonLaunch plan ($19/mo) — always-on compute, branching for preview environments.
FlagsmithSelf-hosted on Northflank with its own Postgres.
Migration effortThree Dockerfile/service configs. Add TRPC_API_URL env var to bot and worker. Swap in-process tRPC calls for HTTP client calls in apps/bot and the worker entry point. No schema changes. No procedure changes. No Redis changes. Estimated: one day.
Migration triggerWhen the bot or worker destabilises the API process, or when independent scaling becomes necessary.
Estimated cost~$80–165/mo depending on Northflank container sizing.

core.guilds — Activity & Data Visibility Columns
The inactivity clock is driven by three columns on core.guilds, each updated independently, plus a generated column that takes the maximum. Falls back to created_at when all three are null — meaning the inactivity clock starts at guild creation for all guilds, including always-free ones. The full core.guilds schema is defined in packages/db — the columns shown here are those with non-obvious runtime behaviour.
-- core.guilds (selected columns — see packages/db for full schema) last_web_activity_at timestamptz -- any tRPC mutation or meaningful web interaction -- debounced: written at most once per hour per guild -- gated via existing cache:guild:{guildId} Redis key last_bot_activity_at timestamptz -- any bot command or Discord event processed -- direct write, no debounce (naturally low frequency) last_raid_event_updated_at timestamptz -- any raid event created or updated -- direct write, no debounce last_plan_change_at timestamptz -- set when plan changes (upgrade or downgrade) -- written by guild.downgrade and supporter link creation -- ensures inactivity clock starts from downgrade time, -- not from the guild's last user activity -- Generated column — do not write directly last_activity_at timestamptz GENERATED ALWAYS AS ( GREATEST( COALESCE(last_web_activity_at, created_at), COALESCE(last_bot_activity_at, created_at), COALESCE(last_raid_event_updated_at, created_at), COALESCE(last_plan_change_at, created_at) ) ) STORED -- Data visibility columns (updated on plan change, new data ingestion, or daily recalculation) data_visible_from timestamptz -- cutoff date for ingested data queries -- set to NOW() - INTERVAL '30 days' at plan-change time for Free -- recalculated daily by data_visible_from.recalculate cron -- to prevent window drift on guilds that stay on the same plan -- NULL for Supporter (no filter applied) has_older_data boolean -- true if ingested data exists beyond data_visible_from -- recalculated synchronously after data_visible_from is written -- also recalculated after each sync.wcl completes -- drives upgrade CTA visibility in the UI
The debounce on last_web_activity_at means it may lag reality by up to one hour. This is intentional and safe — the lag only makes the timestamp appear older than it is, so the archive job would fire later than the true 90-day mark, never earlier. An active guild is never incorrectly archived.
data_visible_from is used as an indexed filter on all ingested data queries: WHERE occurred_at >= guild.data_visible_from. has_older_data is read by the UI to decide whether to show an upgrade CTA — "you have raid data from before your current window, support the project to see it." A daily cron (data_visible_from.recalculate) recalculates and writes this value for all Free-plan guilds — without it, a static timestamp set at plan-change time would drift over time, causing the window to grow rather than slide. Supporter guilds have data_visible_from = NULL and are excluded from this cron.
Multitenancy
Row-level multitenancy with Postgres RLS. Every guild-scoped table carries a guildId column. Cross-cutting tables (core.users, core.user_integrations, internal.*) are not guild-scoped and are exempt from RLS guild isolation. The API service injects guildId from the authenticated session context on every request — it is never accepted from the client payload.
Isolation modelRow-level with RLS enforced at DB layer, not just application layer
guildId sourceSession only — injected by API context factory, never from request body
Cross-guild queriesNot possible from web app or bot. Internal admin and workers can operate cross-guild with elevated service role.
Prisma + RLS implementationPrisma does not natively set Postgres session variables. The standard approach — and what Arel uses — is to wrap every request in a short transaction and run SET LOCAL app.current_guild_id = '...' as the first statement via prisma.$executeRaw. SET LOCAL scopes the variable to the transaction and does not persist across connection reuse — safe with Neon's connection pooler. The context factory (Section 11) is responsible for opening this transaction wrapper on every request before any query runs. Enforcement mechanism: a Prisma client extension ($extends) wraps every query in the transaction and injects the SET LOCAL statement — this ensures no handler can accidentally skip it. The extension is applied once in the context factory; application code never calls $executeRaw or opens the transaction wrapper directly.

Service Identity

Guild roles, service roles, procedure hierarchy, and bot trigger modes.
Every request to the API answers two independent questions: who is the human (user auth), and which system is making the call (service auth). These are orthogonal checks. Service identity determines which procedures a caller is permitted to invoke at all — before any user-level permission check runs.
Guild Role System
Three roles define what a user can do within a guild. Content that requires no login sits outside the guild procedure chain entirely — there is no PUBLIC role, just the absence of one. Routes not wrapped in guildProcedure are unauthenticated by definition.

MEMBER is never stored in Arel — resolved live from Discord OAuth roles at auth time and cached in cache:permissions:{userId}:{guildId} with a 5-minute TTL. A user removed from their Member Discord role loses MEMBER access within 5 minutes with no Arel DB write required. RAID_MANAGER and GUILD_ADMIN are stored in core.guild_seats and enforced explicitly.
First GUILD_ADMIN bootstrap: when a new guild is onboarded, there is no existing GUILD_ADMIN to grant the first seat. Engineering grants the first GUILD_ADMIN seat via the admin UI (internalAdminProcedure) before handing off to the guild. The presumed recipient is the Discord user who authorized the bot OAuth install — confirmed with the guild during onboarding. This action is logged to internal.audit_log. Once the first GUILD_ADMIN seat exists, that user can grant further seats independently. Known scaling constraint: this requires an engineer to act for every new guild signup. At low volume this is manageable; at scale it becomes a bottleneck. The planned automation path is to auto-grant the first GUILD_ADMIN seat to the Discord user who authorized the bot OAuth install at install time — eliminating manual intervention. See Deferred Decisions.
Seat cap enforcement on downgradeWhen guild.downgrade fires and the guild moves to the Free plan (2-seat cap), excess seats are revoked immediately as part of the downgrade job. Seats are revoked in reverse grant order — most recently granted first. GUILD_ADMIN seats are counted against the cap; if total seats exceed 2, RAID_MANAGER seats are revoked first. The guild admin is notified by email listing which seats were revoked and why. Auto-manage cannot re-grant a seat while the guild is at the 2-seat cap — the pending state behaviour applies as normal.

Feature & Plan Gates

Three-axis access control — plan eligibility, feature flags, and usage limits — plus numeric plan benefits, all composed in packages/config.
Three independent axes control access to any given capability. They are composed together in packages/config and enforced via tRPC middleware — never checked inline in route handlers or business logic.
The Three Axes
Plan eligibilityWho: Does this guild's plan include this feature? Resolved from the DB feature group hierarchy. Free features available to all guilds. Supporter features require Supporter plan (plan_id = 'supporter').
Feature flagWhat: Is this feature currently active for this guild? Controlled by Flagsmith — rollout %, kill-switch, scheduled changes, explicit guild overrides. Flagsmith never knows about plans.
Usage limitHow much: Has this guild consumed their quota for this action? DB transaction check — SELECT ... FOR UPDATE, no Redis counters. See Section 10.

Numeric Plan Benefits
Some plan differences are expressed as numeric values rather than feature flags or consumable limits — a number that caps or filters guild behavior based on their plan. These are plan benefits: not counters, not enforced via transactions. The value is defined per plan in config.plan_benefits seed data and overridable per-guild by internal admins via config.guild_config_overrides.
When to useA numeric plan benefit is the right pattern when: the value is a cap or filter applied at read/query time rather than a write-time transaction; there is no natural "one unit consumed" moment. If a value has a consumable counter, use a limit. If it is boolean (on/off), use a feature flag.
config.plan_benefitsSeed data table — one row per benefit_key per plan. benefit_key identifies the benefit (e.g. DATA_VISIBILITY_DAYS). Value is a nullable integer — NULL means unlimited. Changing values per plan requires only a seed data update, no deploy.
Storage strategyTwo approaches depending on access pattern. Static column on core.guilds — value computed at plan-change time and stored directly (e.g. data_visible_from timestamptz). Best for values used as hot-path query filters where a join would be costly. Dynamic resolution — resolved at read time by checking guild_config_overrides then plan_benefits. Best for values checked occasionally (e.g. a max integrations cap checked at connection time). Chosen per benefit, not as a blanket rule.
Per-guild overrideInternal admins set overrides in config.guild_config_overrides using the same config_key as the benefit key. Override persists across plan changes — never auto-recalculated. Resolution: COALESCE(override.value, plan_default). The admin UI shows both values side by side and flags when the override is below the guild's current plan default — divergence is visible and intentional, never silent.
On plan upgradeStatic values are recomputed and written immediately — same step as feature cache bust and limit update. Dynamically resolved values reflect the new plan at next read automatically.
Drift preventionStatic timestamps must be recalculated periodically to prevent window drift. A timestamp set once at plan-change time grows stale — a Free guild that hasn't changed plans in 6 months would have a data_visible_from 6 months ago, not 30 days ago. The data_visible_from.recalculate daily cron recalculates and writes the value for all Free-plan guilds using NOW() - INTERVAL '30 days'. Supporter guilds are excluded — their data_visible_from is always NULL.
On plan downgradeStatic values are recomputed immediately when guild.downgrade fires. Dynamic values reflect the lower plan at next read after the downgrade job completes.

API Service

The API service is the single source of all business logic and data access. It is a standalone Node.js tRPC server running as a persistent container on Northflank. apps/app and apps/bot are consumers — they call procedures, they do not own data.
Why Standalone (Not Next.js Route Handlers)
Connection poolingPersistent container maintains a stable Postgres connection pool. No serverless connection exhaustion. Vercel functions connect to the API over HTTP — they never open a DB connection.
Vercel costVercel functions become thin — receive request, validate session, call API, return response. Minimal duration and compute billed to Vercel.
Shared accessBoth the web app (Vercel) and bot (Northflank) call the same API with the same procedure definitions. Business logic is never duplicated.
No cold startsPersistent container is always warm. No function cold start latency on DB-heavy operations.

Supporter Model

Patreon OAuth per user. Arel verifies pledge status directly — no payment processing in Arel.
How It Works
Connect PatreonUser goes to their personal profile in Arel and connects their Patreon account via OAuth. Arel stores the Patreon access token and member ID against the Arel user, then verifies an active pledge to the Arel campaign via the Patreon API.
Select a guildOn confirmed pledge, the user selects which guild they want to support. One active guild at a time. Selection is locked for the current calendar month — a switch request mid-month writes a pending supporter_links row with effective_month set to the 1st of the following month. The current active link remains in effect until then. On the 1st, the current row is revoked and the pending row becomes active. If the user changes their pending selection before the 1st, the pending row is upserted — last write wins, no additional rows created.
Guild statusGuild plan is stored as plan_id on core.subscriptions and kept in sync by the API service. Multiple supporters can back the same guild — plan_id stays 'supporter' as long as at least one active link exists.
Lapse / cancellationPatreon webhook fires on pledge delete or payment failure. Arel revokes all supporter_links rows for the user immediately — both the current active row and any pending row for next month (identified by revoked_at IS NULL for the user). If no other active links exist for the affected guild, guild.downgrade fires. Nightly patreon.sync polls the Patreon API as a safety net for missed webhooks.
No billing in ArelArel never handles payment details, invoices, or subscription state. All of that lives in Patreon. Arel only verifies pledge status via the Patreon API.
Discord bonusOptionally, Patreon's native Discord integration can assign a supporter role in the Arel Discord server automatically. This is a community perk — not used for verification. Verification is always via Patreon OAuth.

Integrations

External API sync pattern, credential storage, and current integrations.
Integrations pull data from external APIs into the ingested schema. The hot path (user-facing tRPC queries) never calls external APIs directly — it reads from the local copy. All sync is triggered asynchronously via Inngest — never inline in a tRPC handler or bot event.
Sync Pattern
Trigger modelEvent-driven, not polled. Each sync is a discrete action against a specific external resource — triggered by a human action or a bot-detected signal. No background polling loops.
Hot path isolationtRPC queries read only from the local ingested schema. External API calls happen only inside Inngest sync jobs — never on the request path.
DeduplicationEach ingested table carries a UNIQUE constraint on the natural key (e.g. guild_id, wcl_code). Importing the same resource twice is a no-op — upsert on conflict.
ConcurrencyDistributed Redis lock per guild per integration type. Prevents duplicate sync jobs for the same guild running in parallel.
Credential Storage
Per-guild integration credentials are stored in core.guild_integrations — encrypted at the application layer before writing, one row per guild per integration type. The encryption key lives in the API service environment (Northflank env var). No credential is stored in plaintext or in environment variables of other services. Files served directly from R2. Never proxied through Next.js routes. R2 URLs given directly to the client.
07
Redis data is always reconstructible. Nothing stored only in Redis that cannot be rebuilt from Postgres. Redis is a performance layer, not a source of truth.
08
Plan, feature, and limit enforcement is middleware. Never checked inline in route handlers or business logic. The procedure hierarchy is the enforcement point.
09
User-initiated bot commands pass userId. The human is accountable, not the service. Audit log records the user, not "the bot."
10
Every admin action is audit logged. INTERNAL_ADMIN service role triggers audit middleware on every procedure. No exceptions, no opt-outs.
11
Service tokens are DB records. Not hardcoded environment variables. Rotatable and revocable without a redeploy.
12
Plan and limit definitions are seed data. Changing quotas or plan capabilities never requires a code deploy. Update the DB record, bust the cache.
13
Hot path never calls external APIs. tRPC queries read from the local ingested schema. External API calls happen only in Inngest sync jobs.
14
Northflank and Vercel are siblings. They share infrastructure (Postgres, Redis) — they do not call each other for data operations. Coordination via shared DB or event queue only.
15
Integration credentials use application-level encryption. Tokens in core.guild_integrations and core.user_integrations are encrypted by the API service before writing and decrypted after reading. Encryption key stored as an environment variable on Northflank — not in the DB. Never logged, never returned to clients. Key is migratable to a KMS without changing the app interface.
16
Bot user-initiated procedures always resolve the triggering user's Arel role. Never assume permission from Discord role alone. The bot passes triggerUserId and the API resolves their role from the same Redis permission cache as the web client. A MEMBER cannot execute a botRaidManagerProcedure command regardless of what Discord shows them.

Deferred Decisions

Intentionally deferred — foundation is in place to support all of these when the time comes.
To Be Decided
  • Patreon OAuth token storage details — core.user_integrations is defined as the home for Patreon access/refresh tokens, including token_expires_at and status fields (see Section 05). Exact refresh cadence and error recovery flow are to be finalized once the user/guild/event model is complete. The patreon.sync nightly job handles refresh in the interim.
  • Temporary Discord roles for raids — assign a Discord role to signed-up members automatically when a raid is confirmed; remove it after the raid concludes. Supporter feature. Requires Discord OAuth token with role management scope and a background job to clean up roles post-raid.
  • Intelligent signup nudges — automated Discord message triggered via Inngest when signups are below a configurable threshold within a set window before raid start. Supporter feature. Proactive rather than scheduled — reacts to signup state, not a fixed timer.
  • Guild timezone — the "end of day" boundary for same-day event delete logic currently uses UTC (CURRENT_DATE UTC). Guilds raid in their local timezone and a raid at 9pm EST deleted at 8pm EST is "today" in UTC but "not yet started" locally. A guild-configured timezone would make this boundary feel correct to users. Requires a timezone field on core.guilds and replacing CURRENT_DATE checks with (NOW() AT TIME ZONE guild.timezone)::date. Low complexity change — deferred until guilds request it or timezone confusion produces support load.
  • Admin UI IP allowlist — apps/admin should return 404 (not 401) for requests originating outside an allowlisted set of IP addresses. Exact implementation depends on final hosting decision (Northflank vs Vercel).
  • Admin UI deployment target — currently listed as "Northflank or locked-down Vercel project." Decision needed before production. Affects IP allowlist implementation approach.
  • Storage limits — STORAGE_MB removed as an active usage limit type. Replaced by a per-file size cap enforced at upload time. Aggregate storage enforcement deferred until storage costs become a concern.
  • Staging environment — per-PR preview isolation is in place (Section 04). A persistent staging environment (long-lived, production-like, seeded with realistic data) is planned but not yet configured.
  • Admin UI role differentiation — engineering / support / sales roles added after initial flat-role launch
  • First GUILD_ADMIN auto-grant — currently requires manual engineering action per new guild. Planned path: auto-grant the first GUILD_ADMIN seat to the Discord user who authorized the bot OAuth install at install time. Requires verifying the OAuth callback carries sufficient identity context and that the install flow is complete before writing the seat.
  • Pre-implementation blocker. TimescaleDB on Neon availability — verify TimescaleDB extension is available on the chosen Neon plan and region before finalizing hypertable schema. If unavailable, fallback is native Postgres declarative partitioning — architecturally different enough to plan for. Confirm before first migration. usage.usage_events and analytics tables are designed as hypertables; the fallback affects both.
  • Pre-implementation blocker. WCL credentials model — WCL public reports may be fetchable without per-guild auth using a platform-level WCL API key. If so, guilds would not need to connect a WCL account at all — integration setup friction drops to just configuring the watched Discord channel. Requires confirming WCL API access rules for public reports. If private reports need to be supported, per-guild OAuth is required. Decision affects core.guild_integrations WCL row necessity and onboarding flow.
  • Additional external integrations beyond Warcraft Logs — ingested schema pattern supports any API-based source. WCL is cursor-free (report-scoped, idempotent). Future integrations that require a sync cursor (e.g. polling a paginated API from a known offset) will need a cursor column — likely on core.guild_integrations (e.g. sync_cursor jsonb) or a dedicated core.guild_integration_sync_state table. Decision deferred until the first cursor-bearing integration is designed.
  • AI features — pgvector extension on Neon noted as extension point; no current requirement
  • GDPR / right to erasure — guild deletion covers the primary case; individual user erasure flow not yet designed
  • WoW in-game addon architecture — referenced in product but out of scope for this document
  • In-game addon ↔ API communication — will require its own auth token model
  • Search — Postgres FTS to start; Typesense if search performance becomes a concern
  • Public guild API — guild-scoped API keys allowing guild members and developers to build external tooling against Arel data (WeakAuras, spreadsheets, custom scripts). Requires: core.guild_api_keys table with hashed keys, scopes, and per-key rate limiting; a versioned public REST/GraphQL surface separate from the internal tRPC API. Foundation does not block this — no architectural changes needed when ready to build.
